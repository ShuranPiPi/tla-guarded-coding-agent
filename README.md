# TLA+/PlusCal-Guarded Self-Correcting Coding Agent

**CE 356 — Intro to Formal Specification and Verification** Shuran Yan, Youran Ma, Zhuoer Zhang | Northwestern University | Spring 2026

## Overview

This project integrates formal specification components into traditional agentic generation of python flow, as a **Two-Stage Agentic Workflow**. Instead of generating Python code blindly and completely relying on unit tests, our agent first attempts to mathematically specify and verify the algorithm using **PlusCal and TLA+ (TLC Model Checker)**. The reason to do so is following Dijkstra's principles on programming, where abstract thinking and proofs are more important than the actual implementation and execution details. With the integration of formal specification components, we wish to ensure the correctness of algorithms generated by the agent, and improve intepretability of the generation results. 

## Architecture

Our LangGraph state machine operates in two distinct phases:

### Phase 1: Formal Specification & Verification (PlusCal)

The agent breaks down the problem into subtasks, writes PlusCal specifications for each subtask, and checks them using TLC.
```text
┌───────────────────────── LangGraph: Phase 1 ─────────────────────────┐
│                                                                      │
│  Init ──► Plan ──► Split ──► Prove ──► Check ──┬──► (Next Subtask)   │
│                                ▲               │                     │
│                                ├─── Repair ◄───┤ (TLC Errors)        │
│                                │               │                     │
│                                └─── Skip ◄─────┘ (Retry exhausted)   │
└──────────────────────────────────────────────────────────────────────┘
```
(The Skip node implements a critical "Graceful Degradation" mechanism. If TLC verification loops endlessly due to the LLM's limitations with TLA+ syntax, the agent saves the best-effort PlusCal spec as pseudo-code and moves forward, preventing total pipeline failure. This limitation is typical for current LLMs, and the challenges will be addressed in detail in the report)

### Phase 2: Python Code Generation & Execution

The accumulated PlusCal specs (both proved and skipped) are concatenated to serve as a abstract-level plan for the actual Python generation.
```text
┌───────────────────────── LangGraph: Phase 2 ─────────────────────────┐
│                                                                      │
│  (From Phase 1) ──► Generate ──► Test_Python ──┬──► Done             │
│                         ▲                      │                     │
│                         ├─── Repair_Python ◄───┤ (Traceback)         │
│                         │                      │                     │
│                         └─── Fail ◄────────────┘ (Budget spent)      │
└──────────────────────────────────────────────────────────────────────┘
```

### Workflow States
| State | Description |
|-------|-------------|
| **Init / Plan / Split** | Analyze the task and split it into independent sub-tasks (`====SUBTASK====`). |
| **Prove** | Call the LLM to generate a TLA+ module and CFG block containing the PlusCal algorithm for each sub-tasks. |
| **Check** | Automatically invoke the TLC model checker on the generated PlusCal in a local subprocess. |
| **Repair** | If TLC aborts, feed the analyzer log back to the LLM to fix syntax/invariant errors. |
| **Skip** | **[Graceful Degradation]** If PlusCal repair hits `max_retries`, accept the failing spec as pseudocode and continue. |
| **Generate** | Compile all PlusCal specs into a context prompt and translate the logic into idiomatic Python. |
| **Test_Python** | Execute the Python code against assertions in a sandboxed `exec()` environment. |
| **Repair_Python** | Diagnose tracebacks (e.g., type errors, mathematical errors, library version errors...) to patch the Python code. |

## Key Experimental Results & Design Decisions

Through empirical testing, we identified several challenges and developed engineering solutions to address LLM limitations in formal verification:

1. **Overcoming TLA+ Data Imbalance:** LLMs inherently struggle with TLA+'s set-theoretic paradigms and TLC error logs, frequently hallucinating "magic operators" (e.g., trying to call a non-existent `LinearProgrammingSolver` inside TLA+). By introducing the `skip_action`, our system treats failing PlusCal as **high-quality pseudocode blueprints** rather than fatal errors.
2. **Abstract Blueprinting over 1:1 Translation:** In the `Generate` phase, the LLM is explicitly instructed *not* to translate PlusCal line-by-line (which forces useless syntax mimicking). Instead, it extracts the mathematical intent and dynamically selects appropriate Python libraries (e.g., `scipy.optimize.linprog`).
3. **Addressing the "Scope Trap":** The Python execution sandbox was refined to use a unified `local_env` dictionary. This ensures that dynamically imported libraries (like `numpy` or `scipy`) are not lost in the `exec()` closure, preventing misleading `NameError` loops during the repair phase.
4. **Success Cases:** The architecture successfully handles both dynamic data structure problems (e.g., the $O(n \log n)$ `merge_intervals` sort) and complex game-theoretic optimization (e.g., correctly applying sign flips to inequality constraints in the `column_maximin_strategy` zero-sum game).
